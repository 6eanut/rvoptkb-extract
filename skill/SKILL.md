# rvoptkb-extract — RISC-V Optimization Knowledge Extraction Skill

Extracts **Thought** (actionable optimization technique) and **Idea** (general design principle) from a RISC-V optimization commit, following the EoK (Evolution of Kernels) 3-layer knowledge model: **Patch → Thought → Idea**.

The skill manages a **persistent Idea Pool** (`pool/idea-pool.json`) that starts empty and grows organically as patches are processed. Each new patch is matched against existing ideas — if no match is found, a new Idea is added to the pool.

## When to Use

- You have a RISC-V optimization commit/patch and want to extract structured knowledge from it
- You want to build up a knowledge base of optimization techniques organized by RISC-V extensions
- A community interaction agent has detected a new patch merged upstream and needs to record its insights

## Parameters

Single positional argument — path to an input JSON file:

| # | Parameter      | Description                                                                   | Example                                                        |
| - | -------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------------- |
| 1 | `input_path` | Path to input JSON file (produced by `fetch-github.py` or manually crafted) | `oneDNN_bd984d09dc5985a19fb427ac46d19d2cbd5558dd_input.json` |

### Input JSON Format

```json
{
  "patch_subject": "cpu: rv64: gemm: improve gemm kernel performance by tuning LMUL",
  "commit_message": "cpu: rv64: gemm: improve gemm kernel performance by tuning LMUL",
  "code_diff": "diff --git a/src/cpu/rv64/gemm/rvv_gemm_f32.cpp b/src/cpu/rv64/gemm/rvv_gemm_f32.cpp\n@@ -40,19 +40,21 @@ ...",
  "output_path": "/path/to/oneDNN_bd984d09dc5985a19fb427ac46d19d2cbd5558dd_output.json"
}
```

### Output JSON Format (Lightweight)

```json
{
  "patch": {
    "patch_subject": "cpu: rv64: gemm: improve gemm kernel performance by tuning LMUL",
    "commit_message": "...",
    "code_diff": "..."
  },
  "thought": "Use LMUL dynamic tuning to maximize vector register utilization in GEMM kernels on RISC-V V extension hardware",
  "idea": "RVV LMUL Dynamic Tuning in GEMM Kernels"
}
```

> **Note**: The output JSON is intentionally lightweight. For complete information about all ideas in the pool, inspect `pool/idea-pool.json` via `python3 scripts/check-idea-pool.py`.

## Workflow

### Step 0: Determine Project Root

Find the project root directory where `pool/idea-pool.json` and `schema/` live:

```bash
# The SKILL.md target path reveals the project root
realpath "$(dirname "$(realpath ~/.claude/skills/rvoptkb/SKILL.md)")"
# => /home/jiakai/claude-code-workspace/temp/eok/rvoptkb-extract/skill
    PROJECT_ROOT="$(dirname "$PROJECT_ROOT")"
    # => /home/jiakai/claude-code-workspace/temp/eok/rvoptkb-extract
```

Store this as `PROJECT_ROOT`. All relative paths are resolved from this directory.

### Step 1: Read & Validate Input

1. Check that `input_path` (the argument passed to this skill) exists.

   - If the path is relative, try resolving it relative to the current working directory first, then relative to `PROJECT_ROOT`.
   - If still not found, list files matching `*_input.json` in `PROJECT_ROOT` and `PROJECT_ROOT/data/` and prompt the user.
2. Read the input JSON file. Validate that these fields exist and are non-empty:

   - `patch_subject` (string) — first line of the commit message
   - `commit_message` (string) — full commit message
   - `code_diff` (string) — the diff/patch content
   - `output_path` (string) — where to save the output JSON
3. If `output_path` is missing or empty, derive it:

   ```
   dir = directory of input_path (or PROJECT_ROOT if input is in root)
   base = filename without _input.json suffix
   output_path = dir/{base}_output.json
   ```
4. Extract the `source_commit` URL from context:

   - If the input was generated by `fetch-github.py`, derive from the commit SHA in the filename (naming convention: `{repo}_{fullSHA}_input.json`)
   - Construct: `https://github.com/{owner}/{repo}/commit/{SHA}`
   - Note: We may not have `owner` available. If unknown, use the repo name only: `https://github.com/{repo}/commit/{SHA}`
   - If the SHA cannot be extracted from the filename, set `source_commit` to `"unknown"`

### Step 2: Determine if This is an Optimization Commit

Not all RISC-V commits are optimization commits. Before extracting a Thought, check whether this patch truly contains a performance optimization.

**Read the commit message and code diff** to classify the commit:

| Category                   | Indicators                                                                                                                                                                                                                                                                                   | Action                                       |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| **Optimization**     | Performance tuning, speedup, improved throughput, reduced latency; algorithmic improvements (loop unrolling, tiling, vectorization); LMUL tuning; JIT kernel changes; new RVV kernel implementations; parallelism (parallel_nd, OpenMP); data type support enabling performance (FP16, Zvfh) | Proceed to Step 3                            |
| **Non-optimization** | EoL/EOF fixes; copyright header updates; clang-format/whitespace fixes; dead code removal; rebase/merge commits; CI workflow changes (add CI, update CI); documentation-only changes; CODEOWNERS/maintainers changes; pure refactoring with no performance intent                            | **Skip** — see "Skip Procedure" below |

**Skip Procedure:**

If the commit is non-optimization:

1. Print a clear message explaining why this commit was skipped.
2. **Do not** create any Thought or Idea.
3. **Do not** modify the idea pool.
4. **Do not** write an output JSON to `output_path`.
5. Instead, just print a summary and exit.

```
╔══ Skipped (Non-Optimization) ═══════════════════

  Patch:
    Subject: cpu: rv64: binary: add EoL & rebase
    Project: oneDNN

  Reason: This commit is a non-optimization change
          (EoL fix + rebase, no performance impact).

  No output file was written.
  No idea pool changes were made.
═══════════════════════════════════════════════════
```

**Important:** Be thorough in reading the diff content, not just the commit message. A commit titled "fix XYZ" might contain an optimization fix (e.g., fixing a slow code path), while a commit titled "optimize XYZ" might be a trivial whitespace fix. Default to **proceed if unsure**, since false negatives (missed optimizations) are worse than false positives (a non-optimization processed into the pool).

### Step 3: Analyze Patch & Extract Thought

Analyze the commit message and code diff to extract the **Thought**. A Thought is a concise, actionable description of the optimization technique that directly tells an LLM **how** to modify code.

**Read the commit message to understand:**

- What was the problem or performance bottleneck?
- What optimization technique was applied?
- What is the context (project, subsystem, function)?

**Read the code diff to identify:**

- What RISC-V extensions are used?
  - Look for RVV intrinsics: `__riscv_vsetvl`, `__riscv_vadd`, `__riscv_vfmul`, etc. → **V extension**
  - Look for VLA programming patterns (LMUL parameters: `m1`, `m2`, `m4`, `m8`) → **V extension**
  - Look for FP16 types or intrinsics → **Zfh** or **Zvfh** extensions
  - Look for `-march=rv64gcv` or `-march=rv64gc` in build flags → **V extension** or base
  - Look for inline assembly with `.insn` directives
- What data types are involved (FP32, FP16, INT8)?
- What algorithm/loop transformations were done?
- What performance data is mentioned (speedup numbers)?

**Synthesize the Thought** as a single, actionable sentence:

Format: "Use {RISC-V technique} to {achieve optimization goal} in {context}"

Examples:

- "Use LMUL dynamic tuning (m1→m8/m4/m2) to maximize vector register utilization in GEMM kernels on RISC-V V extension hardware"
- "Apply RVV vectorized f32→f64 widening accumulation to maintain numerical precision in group normalization"
- "Leverage Highway SIMD abstraction layer to implement cross-platform vectorized logical operations with RVV backend support"
- "Reduce vector LMUL (m8→m1) to minimize register pressure while introducing thread-level parallelism with parallel_nd"

**Keep track of the thought sentence** — it will be used in the output JSON. Also note the project name and RISC-V extensions for idea matching in Step 3.

### Step 4: Read & Match Against Idea Pool

1. Read the idea pool from `{PROJECT_ROOT}/pool/idea-pool.json`
2. If the file doesn't exist or is malformed (invalid JSON), treat it as an empty pool with `"ideas": []`
3. Compare the extracted thought against existing ideas using **semantic matching**:

**Matching criteria** (use your understanding of the thought's nature):

- **Category overlap**: Does the optimization naturally fall under an existing category?
- **Extension overlap**: Do the RISC-V extensions overlap significantly?
- **Technique similarity**: Is the thought describing a similar class of techniques?

**Decision logic:**

- If **no ideas exist** in the pool → Go to **Step 5** (Create New Idea)
- If a **semantic match is found** (the thought clearly belongs to an existing idea) → reuse the matched idea's title (no pool changes needed)
- If **no match** (the thought describes a fundamentally different optimization direction) → Go to **Step 5** (Create New Idea)

**Important:** Be conservative about matching. It's better to create a slightly overlapping new idea than to force a poor match into an existing one.

### Step 5: Create New Idea (if no match found)

When the thought doesn't match any existing idea:

1. **Generate new idea ID**: `idea-{len(pool['ideas']) + 1:04d}`

   - If pool has 0 ideas → `idea-0001`
   - If pool has 3 ideas → `idea-0004`
2. **Synthesize the Idea** from the Thought — generalize the specific technique into a design principle:

   - **Title**: A concise, general design principle name (5-10 words)
     - Examples: "RVV Vectorized Kernel Optimization", "SIMD Abstraction Layer via Highway", "Build & Runtime Infrastructure Optimization"
   - **RISC-V Extensions**: The extensions this idea relates to
3. **Add the idea** to the pool:

   ```python
   pool["ideas"].append({
     "id": new_id,
     "title": "...",
     "riscv_extensions": [...]
   })
   ```

### Step 6: Save Idea Pool (if modified)

If a new idea was created, write the updated pool back to `{PROJECT_ROOT}/pool/idea-pool.json`:

```python
write_json(pool, pool_path)
```

### Step 7: Generate & Save Output (Lightweight)

Build a concise output JSON:

```python
output = {
    "patch": {
        "patch_subject": input["patch_subject"],
        "commit_message": input["commit_message"],
        "code_diff": input["code_diff"]
    },
    "thought": thought_sentence,    # the synthesized sentence from Step 2
    "idea": idea["title"]         # the matched or new idea's title
}
```

Write to `output_path`. If the directory doesn't exist, try writing to `PROJECT_ROOT` and warn the user.

### Step 8: Report Summary

Print a clear summary of what was done:

```
╔══ Extraction Complete ═══════════════════════════

  Patch:
    Subject: cpu: rv64: gemm: improve gemm kernel performance by tuning LMUL
    Project: oneDNN

  Thought (extracted):
    "Use LMUL dynamic tuning (m1→m8/m4/m2) to maximize vector register
     utilization in GEMM kernels on RISC-V V extension hardware"
    RISC-V Extensions: V

  Idea:
    [NEW] idea-0001: RVV LMUL Dynamic Tuning in GEMM Kernels
      Total ideas in pool: 1

  Output:
    Saved to: /path/to/output.json
    Idea pool: pool/idea-pool.json (1 idea)
```

## Idea Pool Reference

- **Location**: `pool/idea-pool.json` relative to project root
- **Schema**: See `schema/idea-pool-schema.json`
- **Growth**: Data-driven — starts empty, ideas are created as new patch categories are encountered
- **ID Format**: `idea-NNNN` (zero-padded, auto-incremented)
- **Extensions**: Each idea is tagged with relevant RISC-V ISA extensions
- **Inspection**: Use `python3 scripts/check-idea-pool.py` to inspect the pool

## Troubleshooting

| Problem                  | Likely Cause                 | Solution                                                                |
| ------------------------ | ---------------------------- | ----------------------------------------------------------------------- |
| Input JSON not found     | Wrong path or relative path  | Try specifying the absolute path, or place the file in the project root |
| Empty code diff          | Merge commit with no changes | Skip — only commits with code diffs are useful                         |
| Empty idea pool          | First run                    | Expected — the pool grows as patches are analyzed                      |
| Idea pool JSON malformed | Manual edit broke the JSON   | Treat as empty pool, verify the file later                              |
